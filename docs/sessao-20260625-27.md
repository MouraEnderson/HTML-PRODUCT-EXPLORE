## 3. Problemas encontrados e raízes causais

### 3.1 SyntaxError linha 2407 do hotfix
**Causa:** patch anterior introduziu `;` onde deveria ser `,` no `setStatus()`.  
**Efeito:** todo o JS do hotfix falhava silenciosamente. O Netvibes servia versão cacheada sem o bug.

### 3.2 Loading infinito após F5 (10-20 minutos)
**Causa raiz:** `__BOM_WIDGET_BOOT_STATE__` persiste no `window` do Netvibes entre reloads.
```js
// Código problemático:
w.__BOM_WIDGET_BOOT_STATE__ = w.__BOM_WIDGET_BOOT_STATE__ || {...};
// O || preservava started=true, completed=false entre reloads
// executeInit retornava imediatamente sem carregar nada
if (st.started && !st.completed) { return; }
```
**Efeito:** widget ficava em `Loading...` por 10-20 minutos até o Netvibes resetar internamente.

### 3.3 Cache do Netvibes ignorando ?c= na query string
**Causa:** o Netvibes cacheia scripts por nome de arquivo, não por URL completa.  
**Efeito:** mudanças nos arquivos JS não chegavam ao browser mesmo com `?c=novo`.  
**Tentativas fracassadas:** alterar `?c=`, criar arquivos `-r2`, `-fixed`.  
**Solução que funcionou:** remover e readicionar o widget no 3DDashboard (uma vez só).

### 3.4 ProductExplorerBridge = null causou crash
**Causa:** substituição por `null` em vez de `undefined`.  
`typeof null !== 'undefined'` → código tentava chamar métodos em `null` → TypeError.  
**Correção:** usar `undefined` para que os guards `typeof X === 'undefined'` funcionem.

### 3.5 Gráfico de pizza aparecia como barra
**Causa:** seletor `quadCharts` buscava `.bom-charts-row-quad` mas o CSS atual usa `.bom-charts-row`.  
**Correção:** expandir seletor para detectar ambas as classes.

### 3.6 Layout quebrando após mudanças de CSS
**Causa:** CSS tinha `grid-template-columns: repeat(10, ...)` sem `grid-template-areas`.  
O JS (`applyPageGrid` e `apply3dxProductDashboardLayout`) sobrescrevia `gridTemplateRows` quebrando o layout.  
**Correção:** CSS com `grid-template-areas` + neutralizar função JS que sobrescrevia o grid.

### 3.7 Resolução prd-R → dseng quebrada
**Causa:** remoção do `KNOWN_EXPLORER_ROOT_REGISTRY` no sync-provider sem ter o backend Render como substituto funcional.  
**Efeito:** `resolveKnownExplorerRoot` retornava `null` para qualquer projeto, incluindo CJ MESA.  
**Correção:** `resolveKnownExplorerRoot` no hotfix usa fallback interno por título quando o sync-provider retorna null.

### 3.8 Script inline no XHTML causou Loading
**Causa:** tentativa de usar `document.createElement('script')` inline no HTML.  
O Netvibes processa o widget como XHTML 1.0 Strict (XML puro) — script inline com parênteses e ponto quebra o parse XML.  
**Efeito:** widget nunca inicializava.

### Causa raiz do problema persistente de cache
O Netvibes cacheia scripts por nome de arquivo em seu próprio storage interno.  
A única solução definitiva sem mudar o nome dos arquivos é:  
**remover e readicionar o widget no 3DDashboard** (uma vez, recarrega o HTML e os scripts novos).  
Após esse processo, o F5 funciona normalmente (boot timeout de 30s no pior caso).
